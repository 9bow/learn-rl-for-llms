## 도구 호출과 RL의 결합

현대 LLM 에이전트는 다양한 도구(tool)를 호출하여 능력을 확장합니다. 계산기, 코드 실행기, 웹 검색, 데이터베이스 쿼리, API 호출 등이 그 예다. 그러나 도구를 효과적으로 사용하려면 두 가지를 알아야 합니다.

1. **어떤 도구를 사용할 것인가**: 도구 선택
2. **언제 도구를 사용할 것인가**: 도구 호출 시점

기존 연구는 주로 도구 선택에 집중했다. 그러나 최근 연구는 **도구 호출 시점** 이 더 중요하다는 것을 보여줍니다. 같은 도구를 사용하더라도 적절한 시점에 호출하는 것이 핵심입니다.

### 도구 호출의 MDP 모델링

도구 호출을 포함한 에이전트를 MDP로 모델링할 때, 행동 공간은 두 가지 유형을 포함합니다.

$$\mathcal{A} = \mathcal{A}_{language} \cup \mathcal{A}_{tool}$$

**언어 행동** $\mathcal{A}_{language}$: 텍스트 토큰 생성 (추론, 계획 등)
**도구 행동** $\mathcal{A}_{tool}$: 특정 도구 호출 (코드 실행, 웹 검색 등)

RL은 이 두 유형의 행동을 통합하여 최적의 도구 사용 전략을 학습합니다.

## ReTool: 실시간 코드 실행과 언어 추론의 인터리빙

**ReTool** (Feng et al., arXiv:2504.11536)은 언어 추론과 코드 실행을 **동적으로 인터리빙** 하는 전략을 RL로 학습합니다. 핵심 아이디어는 수학 문제 풀기에서 순수 언어 추론이 어려운 부분을 코드 실행으로 보완하는 것입니다.

### 동적 인터리빙

기존 접근법들은 대부분 고정된 방식으로 도구를 사용합니다.

```
[고정 방식] 먼저 코드 작성, 그 다음 언어 추론
[ReTool] 언어 추론 ↔ 코드 실행을 동적으로 전환
```

ReTool의 에이전트는 다음을 스스로 결정합니다.
- 지금 이 시점에 코드를 실행해야 하는가?
- 코드 실행 결과를 어떻게 해석하고 다음 추론에 반영할 것인가?
- 언제 코드 없이 순수 언어 추론만으로 충분한가?

### RL 학습 목표

ReTool의 정책은 다음을 최적화합니다.

$$\pi^*(a_t | s_t) = \arg\max_\pi \mathbb{E}\left[r_{final} + \lambda \cdot r_{efficiency}\right]$$

여기서 $r_{final}$ 은 최종 정답 여부, $r_{efficiency}$ 는 불필요한 도구 호출 페널티다.

$$r_{efficiency} = -\alpha \cdot \text{count}(\text{unnecessary\_tool\_calls})$$

이 설계는 모델이 **필요할 때만** 코드를 실행하도록 학습시킨다.

### AIME 성능

ReTool은 Qwen2.5-32B를 기반으로 다음 성능을 달성했다.

| 벤치마크 | 기준 SFT | 코드만 사용 | **ReTool (RL)** |
|---------|---------|----------|----------------|
| AIME 2024 | 51.3% | 58.7% | **67.0%** |
| AIME 2025 | 38.4% | 43.2% | **49.3%** |
| MATH-500 | 88.1% | 89.4% | **93.2%** |

순수 코드 사용 대비 AIME 2024에서 8.3%p 추가 향상은 인터리빙 전략의 효과를 명확히 보여줍니다.

### 왜 인터리빙이 중요한가

수학 문제의 각 단계는 다른 특성을 갖습니다.

| 단계 유형 | 최적 접근법 |
|---------|----------|
| 개념 이해, 전략 선택 | 언어 추론 |
| 복잡한 수치 계산 | 코드 실행 |
| 경우의 수 열거 | 코드 실행 |
| 결과 해석, 검증 | 언어 추론 |
| 예외 처리, 엣지 케이스 | 인터리빙 |

RL은 각 단계의 특성을 학습하여 최적 접근법을 자동으로 선택합니다.

## 멀티스텝 도구 오케스트레이션

실제 에이전트 태스크에서는 단일 도구 호출이 아니라 여러 도구를 순서대로 호출하는 **멀티스텝 오케스트레이션** 이 필요합니다.

### 도전 과제

멀티스텝 도구 사용에는 추가적인 어려움이 있습니다.

- **도구 간 의존성**: 이전 도구의 출력이 다음 도구의 입력
- **오류 전파**: 초기 도구 호출 실패가 전체 체인에 영향
- **보상 희소성**: 전체 체인 완료 후에만 보상

### 제약 데이터 합성

멀티스텝 도구 오케스트레이션을 위한 훈련 데이터 수집은 어렵습니다. 인간 전문가가 복잡한 도구 체인을 시연하는 것은 매우 비용이 높습니다. **제약 데이터 합성(constrained data synthesis)** 은 이 문제를 해결합니다.

규칙 기반 제약을 활용하여 자동으로 유효한 도구 호출 시퀀스를 생성합니다.

```
제약 정의:
- 도구 A의 출력은 도구 B의 입력 유형과 일치해야 함
- 도구 C는 도구 D 이후에만 호출 가능
- 최대 10개의 도구 호출 허용

자동 생성:
1. 제약을 만족하는 도구 체인 샘플링
2. 각 체인에 대한 실행 결과 수집
3. 성공한 체인만 훈련 데이터로 사용
```

### 단계적 보상 구조

멀티스텝 오케스트레이션에서는 **단계적 보상** 이 효과적입니다.

$$r_{total} = r_{final} + \sum_{k=1}^{K} w_k \cdot r_k$$

여기서 $r_k$ 는 $k$ 번째 도구 호출 단계의 중간 보상입니다. 이전 도구의 출력이 유효한지, 다음 단계로 올바르게 전달되는지를 평가합니다.

## RLFactory: 플러그앤플레이 RL 후훈련 프레임워크

**RLFactory** 는 도구 사용 RL을 다양한 LLM에 쉽게 적용할 수 있는 **플러그앤플레이 프레임워크** 다.

### 설계 원칙

RLFactory는 세 가지 원칙으로 설계되었다.

**모듈성**: 도구 정의, 보상 함수, RL 알고리즘을 독립적으로 교체 가능

**확장성**: 새로운 도구를 추가해도 기존 훈련 파이프라인 수정 불필요

**이식성**: 다양한 LLM 아키텍처에 적용 가능

### 지원 도구 유형

| 도구 유형 | 예시 | 보상 정의 |
|---------|------|---------|
| 코드 실행기 | Python, JavaScript | 실행 결과 정확성 |
| 웹 검색 | SerpAPI, Bing | 관련성 점수 |
| 계산기 | 수식 계산 | 수치 정확성 |
| 데이터베이스 | SQL 쿼리 | 쿼리 결과 정확성 |
| API 호출 | REST API | 상태 코드 및 응답 |
| 파일 시스템 | 파일 읽기/쓰기 | 작업 성공 여부 |

### 훈련 파이프라인

```
1. 도구 정의: YAML로 도구 스펙 작성
2. 보상 함수: 도구별 보상 계산 로직 정의
3. 데이터 생성: 제약 기반 자동 데이터 합성
4. RL 훈련: GRPO/PPO 선택적 적용
5. 평가: 도구별, 태스크별 성능 측정
```

## 언제 도구를 쓸지 vs 어떤 도구를 쓸지

도구 사용 RL의 가장 중요한 통찰은 **도구 호출 시점** 이 **도구 선택** 보다 더 중요하다는 것입니다.

### 왜 시점이 중요한가

같은 코드 실행 도구를 사용하더라도 다음과 같은 차이가 발생합니다.

| 시점 선택 | 결과 |
|---------|------|
| 너무 이른 코드 실행 | 문제 이해 전 코드 작성 → 잘못된 알고리즘 |
| 너무 늦은 코드 실행 | 언어 추론으로 처리 가능한 부분도 느리게 처리 |
| 적절한 시점 | 언어 추론과 코드 실행의 강점을 각각 활용 |

RL은 이 최적 시점을 수천 번의 시행착오를 통해 자동으로 학습합니다. 인간이 규칙으로 정의하기 어려운 이 미묘한 판단을 RL이 데이터로부터 학습하는 것이 핵심입니다.

### 형식 보상 설계

도구 호출 시점 학습을 위해 특별한 형식 보상이 도움이 됩니다.

$$r_{format} = \begin{cases} +0.2 & \text{올바른 도구 호출 형식} \\ -0.5 & \text{불필요한 도구 호출} \\ -0.3 & \text{도구 필요 시 미호출} \end{cases}$$

## 정리

도구 호출 RL은 LLM이 외부 도구를 전략적으로 활용하는 능력을 근본적으로 향상시킨다. ReTool은 언어 추론과 코드 실행의 동적 인터리빙을 RL로 학습하여 AIME 2024에서 67.0%를 달성했다. RLFactory는 이 접근법을 다양한 도구와 LLM에 쉽게 적용할 수 있는 프레임워크를 제공합니다. 핵심 통찰은 "어떤 도구를 쓸지"보다 "언제 도구를 쓸지"가 더 중요하며, 이 판단을 RL이 효과적으로 학습할 수 있다는 것입니다.
