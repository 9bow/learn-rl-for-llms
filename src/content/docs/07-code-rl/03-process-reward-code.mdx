## 프로세스 보상 모델의 필요성

코드 생성에서 **결과 기반 보상(Outcome Reward, ORM)** 은 직관적입니다. 코드가 테스트를 통과하면 $r=1$, 아니면 $r=0$입니다. 그러나 이 단순함은 심각한 한계를 낳는다.

긴 알고리즘을 구현할 때 모델이 올바른 방향으로 추론하다가 마지막 단계에서 실수를 하더라도, ORM은 전체를 실패로 처리합니다. 반대로 단순 해킹(예: 테스트 케이스 하드코딩)으로 테스트를 통과한 코드는 성공으로 처리합니다. **프로세스 보상 모델(Process Reward Model, PRM)** 은 이 문제를 생성 과정의 각 단계를 평가함으로써 해결합니다.

$$r_{PRM}(c) = \sum_{i=1}^{N} w_i \cdot r(s_i)$$

여기서 $s_i$ 는 코드 생성의 $i$ 번째 단계(예: 논리 블록, 함수 구현 단계), $w_i$ 는 가중치다.

## ThinkPRM: 추론하는 프로세스 보상 모델

**ThinkPRM** (Xia et al., 2025)은 프로세스 보상 모델이 단순히 점수를 매기는 것이 아니라, LLM처럼 **추론을 수행** 한 후 점수를 산출해야 한다는 핵심 통찰에서 출발합니다.

### 기존 PRM의 한계

기존 PRM은 각 단계 $s_i$ 에 대해 직접 스칼라 점수를 출력합니다. 이는 빠르지만, 복잡한 추론 단계의 정확성을 판단하는 데 필요한 **내부 추론 과정** 이 없습니다.

### ThinkPRM의 접근법

ThinkPRM은 각 단계를 평가하기 전에 Chain-of-Thought 스타일의 내부 추론을 수행합니다.

```
[단계 i 평가]
<think>
이 코드 블록은 배열을 정렬한 후 이진 탐색을 수행한다.
정렬은 O(n log n)이고 올바르다. 이진 탐색의 경계 조건을
확인하면 lo <= hi 조건이 적절하다. 반환값도 -1로 올바르다.
</think>
점수: 0.92
```

이를 통해 ThinkPRM은 단순한 패턴 매칭이 아닌 실제 논리적 검증을 수행합니다.

### 벤치마크 성능

ThinkPRM은 **ProcessBench** (단계별 추론 오류 탐지 벤치마크), MATH-500, AIME'24에서 기존 검증기 대비 큰 폭의 성능 향상을 달성했다.

| 벤치마크 | 기존 최고 ORM | 기존 최고 PRM | ThinkPRM | 향상 |
|---------|-------------|-------------|---------|------|
| ProcessBench | 71.2% | 74.8% | **82.9%** | +8.1%p |
| MATH-500 | 88.4% | 90.1% | **93.7%** | +3.6%p |
| AIME'24 | 52.3% | 55.7% | **63.4%** | +7.7%p |
| GPQA-Diamond | 48.9% | 51.2% | **57.8%** | +6.6%p |

기존 검증기 대비 평균 **8% 이상** 의 성능 초과는 "추론하는 검증기"의 유효성을 강력히 지지합니다.

## FunPRM: 함수 수준 프로세스 보상

**FunPRM** 은 코드의 자연스러운 구조 단위인 **함수** 를 프로세스 보상의 기본 단위로 삼는다. 복잡한 프로그램은 여러 함수의 조합이므로, 함수별로 보상을 분리하면 더 정확한 크레딧 할당이 가능합니다.

### 함수 수준 분해

프로그램 $P = \{f_1, f_2, \ldots, f_m\}$ 으로 분해할 때, FunPRM은 각 함수 $f_i$ 에 대해 독립적인 보상을 계산합니다.

$$r_{FunPRM}(P) = \prod_{i=1}^{m} r(f_i)^{\alpha_i}$$

여기서 $\alpha_i$ 는 함수 $f_i$ 의 복잡도 기반 가중치다.

### 메타 보상 교정

FunPRM의 독특한 기여는 **메타 보상 교정(meta reward calibration)** 입니다. PRM이 잘못 보정(miscalibrated)되어 있을 경우, 이를 자동으로 감지하고 교정하는 메커니즘입니다.

교정 전 PRM 점수 $\hat{r}$ 과 실제 실행 결과 $r_{exec}$ 의 불일치를 측정하여 교정 함수 $g$ 를 학습합니다.

$$r_{calibrated} = g(\hat{r}, \text{context})$$

이를 통해 특정 코드 패턴에 과적합된 PRM의 편향을 줄입니다.

## R-PRM: 추론 과정 기반 보상

**R-PRM** 은 코드 생성 과정에서 모델이 작성하는 **자연어 추론(rationale)** 을 보상 신호의 일부로 활용합니다. 코드와 함께 생성된 주석, 설명, 중간 계획이 얼마나 논리적인지를 평가합니다.

$$r_{RPRM}(c, \text{rationale}) = \lambda \cdot r_{exec}(c) + (1-\lambda) \cdot r_{logic}(\text{rationale})$$

이 접근법은 "올바른 코드"뿐만 아니라 "올바른 이유로 생성된 코드"를 장려합니다.

## Process Supervision-Guided Policy Optimization

Wang et al. (arXiv:2410.17621)은 PRM을 정책 최적화에 직접 통합하는 프레임워크를 제안합니다. PRM 점수를 단순히 최종 선택에만 사용하는 것이 아니라, **훈련 중 단계별 정책 경사 신호** 로 활용합니다.

$$\nabla_\theta \mathcal{L} = -\sum_{t=1}^{T} r_{PRM}(s_t) \cdot \nabla_\theta \log \pi_\theta(a_t | s_t)$$

이를 통해 모델은 높은 PRM 점수를 받는 생성 경로를 더 자주 선택하도록 학습됩니다.

## ORM vs PRM 코드 비교

| 특성 | ORM (결과 기반) | PRM (프로세스 기반) |
|------|---------------|------------------|
| 보상 시점 | 코드 실행 후 | 각 생성 단계마다 |
| 보상 희소성 | 높음 | 낮음 |
| 보상 해킹 취약성 | 높음 (하드코딩) | 낮음 |
| 크레딧 할당 | 어려움 | 정확함 |
| 계산 비용 | 낮음 | 높음 |
| 구현 복잡도 | 낮음 | 높음 |
| 부분 정답 처리 | 불가 | 가능 |
| 훈련 데이터 요구 | 적음 | 많음 (단계 레이블) |
| 코드 도메인 적용 | 자연스러움 | 구조 경계 활용 |

## 코드에서의 PRM 특수성

코드 도메인은 수학 추론과 달리 **명확한 프로세스 경계** 를 제공한다는 장점이 있습니다. 수학에서는 "단계"의 정의가 모호하지만, 코드에서는 다음과 같은 자연스러운 경계가 존재합니다.

- 함수 정의 경계: `def`, `function`, `fn`
- 제어 흐름 블록: `if/else`, `for`, `while`
- 표현식 단위: 단일 문장
- 주석 구분: 개발자가 명시한 논리 단계

이러한 구조적 특성 덕분에 코드에서의 PRM 레이블 생성은 수동 주석 비용을 낮출 수 있습니다. 컴파일러가 각 블록의 중간 상태를 추적하고 검증할 수 있기 때문입니다.

## 정리

프로세스 보상 모델은 코드 생성의 품질을 결과뿐만 아니라 과정에서 평가함으로써 희소 보상과 보상 해킹 문제를 완화합니다. ThinkPRM의 추론 기반 검증, FunPRM의 함수 수준 분해와 메타 교정, R-PRM의 추론 품질 통합은 각기 다른 각도에서 프로세스 보상의 가능성을 확장하고 있습니다. 코드의 구조적 명확성은 PRM 적용에 특히 유리한 환경을 제공합니다.
