## 왜 코드 생성은 RL에 적합한가

코드 생성(code generation)은 강화학습(RL)의 가장 자연스러운 적용 영역 중 하나다. 그 핵심 이유는 **검증의 자동화** 에 있습니다. 자연어 생성에서는 "좋은 문장"인지 판단하려면 인간 평가자나 복잡한 보상 모델이 필요하지만, 코드의 경우 컴파일러와 인터프리터가 즉각적이고 정확한 피드백을 제공합니다.

### 자동 검증기로서의 컴파일러

Python, Java, C++ 등 현대 프로그래밍 언어의 실행 환경은 생성된 코드에 대해 다음과 같은 신호를 자동으로 반환합니다.

| 피드백 유형 | 신호 | 보상 매핑 |
|------------|------|----------|
| 컴파일/파싱 성공 | 구문 오류 없음 | $r_{syntax} > 0$ |
| 런타임 실행 성공 | 예외 없음 | $r_{runtime} > 0$ |
| 단위 테스트 통과 | assert 성공 | $r_{test} \in \{0, 1\}$ |
| 기능적 정확성 | 모든 테스트 통과 | $r_{functional} = 1$ |

이러한 계층적 피드백은 RL의 희소 보상(sparse reward) 문제를 자연스럽게 완화합니다. 완전한 정답이 아닌 중간 단계에서도 의미 있는 보상 신호를 추출할 수 있기 때문입니다.

### 테스트 케이스 기반 보상 설계

테스트 케이스 $\mathcal{T} = \{(x_1, y_1), \ldots, (x_n, y_n)\}$ 가 주어질 때, 생성된 코드 $c$ 의 보상은 다음과 같이 정의됩니다.

$$r(c, \mathcal{T}) = \frac{1}{|\mathcal{T}|} \sum_{i=1}^{|\mathcal{T}|} \mathbf{1}[\text{execute}(c, x_i) = y_i]$$

이 공식은 단순하지만 강력하다. 모델은 "어떤 코드가 좋아 보이는가"가 아니라 "어떤 코드가 실제로 작동하는가"를 학습하게 됩니다.

## CodeRL: Actor-Critic 프레임워크

**CodeRL** (Le et al., 2022)은 코드 생성을 위한 최초의 체계적인 RL 프레임워크 중 하나다. CodeT5를 기반으로 하며, Actor-Critic 구조를 통해 단위 테스트 피드백을 효과적으로 활용합니다.

### 아키텍처 구성

```
Program Synthesis Problem
        ↓
   Actor (CodeT5)
        ↓ 코드 생성
   Unit Test Executor
        ↓ 실행 결과
   Critic Model
        ↓ 오류 분류 및 가치 추정
   Policy Gradient Update
```

**Actor** 는 사전훈련된 CodeT5 모델로, 주어진 문제 명세(problem specification)로부터 코드를 생성합니다. **Critic** 은 별도로 훈련된 모델로, 생성된 코드 토큰 시퀀스를 분석하여 각 상태의 가치 $V(s)$ 를 추정합니다.

### 비평 모델의 오류 분류

Critic 모델은 단순히 "성공/실패"를 판단하는 것이 아니라 오류를 세밀하게 분류합니다.

| 오류 클래스 | 설명 | 보상 값 |
|------------|------|--------|
| `PASS` | 모든 테스트 통과 | $+1.0$ |
| `WRONG_ANSWER` | 실행은 되지만 결과 불일치 | $-0.3$ |
| `SYNTAX_ERROR` | 파싱 불가 | $-0.6$ |
| `RUNTIME_ERROR` | 실행 중 예외 발생 | $-0.5$ |
| `TIME_LIMIT` | 시간 초과 | $-0.4$ |
| `COMPILE_ERROR` | 컴파일 실패 | $-0.6$ |

이 세밀한 분류는 Actor가 어떤 유형의 오류를 줄여야 하는지 방향성을 제공합니다.

### 학습 목표

CodeRL의 정책 경사 목표는 다음과 같습니다.

$$\mathcal{L}_{RL} = -\mathbb{E}_{c \sim \pi_\theta} \left[ \sum_{t=1}^{|c|} (r(c) - V_\phi(s_t)) \cdot \log \pi_\theta(c_t | c_{<t}, x) \right]$$

여기서 $V_\phi(s_t)$ 는 Critic이 추정한 기준선(baseline)으로, 분산을 줄이는 역할을 합니다. 전체 손실은 지도학습 손실과 RL 손실의 가중 합으로 구성됩니다.

$$\mathcal{L} = \mathcal{L}_{SL} + \lambda \mathcal{L}_{RL}$$

## 코드 RL의 고유한 도전 과제

코드 생성을 위한 RL은 일반적인 언어 생성 RL과 구별되는 독특한 어려움을 갖습니다.

### 1. 부분 정답(Partial Correctness) 문제

$n$ 개의 테스트 케이스 중 $k$ 개를 통과한 코드는 어떤 보상을 받아야 하는가? 이진 보상(0 또는 1)은 너무 희소하고, 연속 보상(통과 비율)은 잘못된 코드에도 보상을 줄 수 있습니다. 연구자들은 숨겨진 테스트(hidden test)와 공개 테스트(public test)의 혼합 전략으로 이를 완화합니다.

### 2. 실행 시간 제약

코드 실행 환경을 RL 훈련 루프에 통합하면 상당한 지연(latency)이 발생합니다. 수백만 번의 정책 평가를 위해 매번 코드를 실제로 실행하는 것은 비현실적입니다. 따라서 **비평 모델로 실행 결과를 예측** 하거나 **비동기 실행 파이프라인** 을 구축하는 방식이 필요합니다.

### 3. 보안 및 샌드박스 이슈

훈련 중 LLM이 생성하는 코드는 임의의 시스템 명령, 파일 삭제, 네트워크 접근 등 위험한 연산을 포함할 수 있습니다. 실제 코드 RL 시스템은 격리된 도커 컨테이너나 가상 머신 환경에서 코드를 실행해야 합니다.

### 4. 보상 해킹(Reward Hacking)

모델은 테스트를 "통과"하기 위해 하드코딩이나 테스트 케이스 역추적 같은 의도치 않은 전략을 학습할 수 있습니다. 예를 들어 공개된 예시 입출력만 외워서 반환하는 식입니다. 이는 충분한 수의 숨겨진 테스트와 다양한 엣지 케이스로 방어해야 합니다.

## 성능 비교

CodeRL은 APPS 벤치마크(Hendrycks et al., 2021)에서 다음과 같은 성능을 달성했다.

| 모델 | Introductory | Interview | Competition |
|------|-------------|-----------|-------------|
| GPT-Neo 2.7B | 3.90% | 0.57% | 0.00% |
| CodeT5 (SFT) | 7.25% | 1.15% | 0.10% |
| **CodeRL** | **11.41%** | **1.91%** | **0.20%** |

RL 훈련을 통해 기존 지도학습 대비 약 57% 향상을 달성했다. 이는 코드 도메인에서 RL의 유효성을 처음으로 체계적으로 입증한 결과입니다.

## 정리

코드 생성은 컴파일러와 테스트 케이스라는 자동화된 검증 메커니즘 덕분에 RL에 이상적으로 적합하다. CodeRL은 이를 Actor-Critic 프레임워크로 구체화하여 단위 테스트 피드백을 세밀한 보상 신호로 변환했다. 그러나 부분 정답, 실행 지연, 보안 문제, 보상 해킹 등의 도전 과제는 이후 연구들이 해결해야 할 중요한 과제로 남아 있습니다.
